# 平衡车修复计划（低频抖 / 速度环无力 / 自动中值）

研究 + 代码审计得出。**按顺序执行,每步单独验证,不要跳步。** 全程留好"扶住/断电"准备。

## 诊断（根因）
- **低频抖,Kp/Kd 治不了** = 电机**死区极限环**(非线性)。Kp/Kd 只缩放线性部分,注不进死区里缺的扭矩 → 永远治不掉,只改变抖的幅度/频率。
- **推不停 / 推不回** = 我加的 `VEL_OUT_LIMIT=±1800` 把整个速度环输出卡死在半量程。**无论 vKp 多大,最多只出 1800**(满量程 3600)。+ OLED 只有 4 位,≥10000 显示成 0,让你以为 vKp 归零了(其实 int16 到 32767 都有效)。
- **按 M 就失控**(真凶,新发现) = **单位 bug**:角度环把 `ax_middle_angle` 当**度**用(`target - balance_angle`,而 balance_angle 是度),但 M 存进去的是 **0.01 度**。按 M 设成 -100(=-1°)→ 角度环理解成 **-100° 目标** → 全功率冲 → 失控。FallProtect/PutDown 又按 0.01° 解读,三处不一致。

---

## Step 1 — 修 OLED 4 位显示（先能看清数值）
`ax_task.c`：line97 `OLED_ShowNum(4,4,ax_velocity_kp,4)`→宽度 **5**；line99 `OLED_ShowNum(4,12,ax_velocity_ki,4)`→**5**；line90(可选)`ax_balance_kp`→**5**。
**验证**:vKp 加过 10000,OLED 显示 `10200` 而非 `0`。确认 16 列排版不撞(vP:1-3,数 4-8,vI:9-11,数 12-16)。

## Step 2 — 解除速度环钳位（治"推不停",最高优先）
`ax_balance.c` line39：`VEL_OUT_LIMIT 1800` → **3000**(留 600 给角度环;还弱再上 3600)。
**为什么**:速度环靠"动态倾角偏置"来刹车/回位,那个偏置就是 ax_velocity_out;±1800 把它卡在半量程,所以 vKp=9800 没用——输出早顶在 1800。
**验证**(架空,V 开 T 关 vKi=0):拨轮 → ch5 现在能冲到 ±3000(原来顶在 1800)。落地推车 → 能减速停住。若顺手冲飞=符号反,按一次 `Y`(别再加钳位)。

## Step 3 — 恢复积分回位力（治"推不回"）
`ax_balance.c` line166-167(积分钳 ±0.8）。**Step 2 验证后再做**,二选一:
- **A(先试这个)**:vKi 用蓝牙降回 ~**4000**,积分钳保持 0.8(4000×0.8=3200<量程)。之前 9900 没用是被 1800 压死,不是积分太小。
- **B(A 还不够)**:vKi 用中等值,积分钳放宽到 ±1.5(保持 vKi×钳<~3200,如 vKi2000×1.5=3000)。
**验证**:推车 ~20cm 松手 → 自己**往回滚到起点**附近停。ch5 回 ~0。过冲就降 vKi。

## Step 4 — 电机死区补偿（治低频抖的根）
先**测**:架空、环全关,从 0 慢慢加裸 PWM,记每个轮**刚开始转**的值(约 300-550)= DZ_L/DZ_R,在 line41 附近加 `#define`。
`ax_balance.c` ISR 内,混合之后(line102-103)、写电机之前(line107)插入:
```c
if      (wheel_pwm_l > 0) wheel_pwm_l += DZ_L;
else if (wheel_pwm_l < 0) wheel_pwm_l -= DZ_L;
if      (wheel_pwm_r > 0) wheel_pwm_r += DZ_R;
else if (wheel_pwm_r < 0) wheel_pwm_r -= DZ_R;
/* 加完再钳回 ±3600 */
```
(pwm==0 不动,让它能静止;必须加在**最终混合 PWM** 上,不是每个环。)
**验证**:0–1° 慢摆明显变小/消失。**调法**:DZ 加到车静止刚开始抽搐,再退到不抽。每轮单独标定。

## Step 5 — 修 midpoint 单位 bug（M 能用的前提,根治"按M失控"）
`ax_balance.c`:把 `Balance_AngleCtl` 里 line131 改成 `angle_bias = (target * 0.01f) - angle;`(target 仍传 raw 0.01°)。这样 line131(度)、line222(PutDown)、line249(FallProtect)统一按 0.01° 读,和 OLED 标签一致。
**验证**:midpoint=0 时行为不变(回归)。按 M 设 50(=0.5°)→ 平衡点只偏**半度**,不是 50°;FallProtect 仍在新中值 ±45° 跳。

## Step 6 — 带门控的自动中值（你要的功能）
复用 `ax_robot.c` line28 `ax_midcal_enable`(默认 0),`ax_control.c` 加一个蓝牙键开关,`ax_balance.c` 加 `Balance_AutoMidpoint()` 在 ISR 老位置(line118 与 Balance_PutDown 之间)调用。
**逻辑(取震荡中心,不是自由积分)**:维护 `ax_balance_angle` 的滑动 min/max(约 1-2s 窗口),仅当**全部门控成立**时把 `ax_middle_angle` **缓慢**移向 `(min+max)/2`:
1) move_enable=1 2) midcal_enable=1 3) |gyro| 小 4) 窗口幅度<±2° 5) 轮速很小 6) vx=vw=0 7) ax_velocity_out 没饱和。
任一门控掉→重置窗口;`ax_middle_angle` 硬钳 ±300(±3°)。
**为什么不会像上次跑飞**:上次是 `midpoint += (angle-midpoint)` 无界自由积分,失稳时一路跑偏;现在是**有界窗口的中心**(不可能超出窗口边),且严格门控(推/摔时全部门控为假→冻结而非漂移)+ 限速 + 硬钳 ±3°。
**验证**:开启后稳定小摆时 OLED `M:` 几秒内缓慢移到摆动中心(如 ~50)、抖动收敛。**安全测试**:用力推 / 抱起 → `M:` 必须**冻结**不动。掉了 → 窗口重置。若推时 M 动,收紧门控 3/5/7。

**顺序**:Step1(可读)→ 2(推得停)→ 3(推得回)→ 4(不抖)→ 5(M 单位)→ 6(自动中值)。
