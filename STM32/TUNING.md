## 1. 原理：自平衡小车是什么 + 三环串级控制

### 1.1 倒立摆（inverted pendulum）

自平衡小车本质是一个**倒立摆（inverted pendulum）**：质心在轮轴**上方**，所以它天生不稳定 —— 只要稍微偏离竖直，重力就会产生一个**越来越大**的倾倒力矩，越倒越快。它不像普通钟摆会自己回到竖直位置；它必须**靠主动控制**才能站住。

控制的核心思想很简单：**车往哪边倒，轮子就往哪边走，把支点（轮子）移到质心下方，重新接住它。** 就像你用手掌顶一根扫帚 —— 扫帚往前倒，你的手就要往前移。

要做好这件事，光「接住倾倒」还不够。一个真正能长期站稳、还能听指令前进/转向的小车，需要**三个**控制环协同工作。本固件用的是**串级（cascaded）三环**结构，全部跑在 100 Hz。

### 1.2 控制循环跑在哪里、跑多快

平衡控制**不在** FreeRTOS 任务里，而在 **MPU6050 数据就绪中断**里。MPU6050 的 DMP 每 10 ms（100 Hz）在 PA5 上拉一个中断（EXTI Line 5），触发 `EXTI9_5_IRQHandler()`（`ax_balance.c:40`，处理函数判断 `EXTI_GetITStatus(EXTI_Line5)`）。这样做的好处是：**无论任务怎么调度，平衡环都以硬件定时的 100 Hz 雷打不动地运行**（`ax_task.c:5-6` 也强调了这一点）。

每次中断里发生的事（`ax_balance.c:52-92`）：

1. 通过 DMP 读 IMU：
   - `ax_balance_angle = ax_angle_data[1]` —— **PITCH**（绕 Y 轴的前后倾角），单位 0.01°（`ax_balance.c:53`）。
   - `ax_balance_gyro = ax_gyro_data[1]` —— Y 轴角速度原始值（`:54`）。
   - `ax_turn_gyro = ax_gyro_data[2]` —— Z 轴（yaw 偏航）角速度原始值（`:55`）。
2. 换算成工程单位：`balance_angle`(deg)、`balance_gyro`(deg/s, ×0.061)、`turn_gyro`(deg/s)（`:57-59`）。
3. 读编码器算左右轮速（m/s；注意本车**物理前进读负** —— 这是速度环需要的符号，详见 Phase 2），读完清零。
4. 跑**三个环**，再混合成左右 PWM，最后送电机（详见下文）。

> 注：这里的轴选择（用 PITCH `[1]`）是本车相对厂商 OpenCTR 原版（用 `[0]`）唯一改动之一，因为本车前后倾是绕 Y 轴的（`ax_balance.c:7-9`）。

### 1.3 三个环分别是什么

#### 环 1：角度环（ANGLE loop）= PD —— 把车「扶直」

**测量什么**：倾角 `balance_angle`（来自 DMP pitch）和倾角速率 `balance_gyro`（来自陀螺仪 Y 轴）。
**输出什么**：`ax_balance_out`，一个让车「往倒的方向追、把自己接住」的基础 PWM。
**为什么需要**：这是最核心、最先要调好的环。没有它，车立刻倒。

代码（`Balance_AngleCtl()`，`ax_balance.c:113-126`）：

```c
angle_bias = target - angle;        // target = ax_middle_angle（中值），即「离平衡点差多少」
gyro_bias  = 0 - gyro;              // 期望角速度是 0（不希望它在转）
p_out = ax_balance_kp * angle_bias * 0.1f;   // P：偏差越大，纠正越猛
d_out = ax_balance_kd * gyro_bias  * 0.1f;   // D：倒得越快，越用力刹（阻尼）
out = p_out + d_out;
```

- **`ax_balance_kp`（balance Kp）**：比例增益。决定「偏离中值多少度，就给多大纠正力」。Kp 太小车扶不住会慢慢倒；太大则反应过激、来回猛冲，开始**高频抖动 / 振荡**。
- **`ax_balance_kd`（balance Kd）**：微分增益，作用在**角速度**上（注意是 `0 - gyro`）。它是**阻尼**：车一旦开始快速倾倒/快速回摆，D 项立刻反向用力「踩刹车」，抑制过冲和振荡。Kd 太小会过冲、低频晃；太大则会放大传感器噪声、引入**高频小抖动 / 电机嗡嗒响**。

> 这是一个 **PD** 环：只有 P 和 D，**没有 I**。角度环故意不积分（积分会让它对中值误差死磕、容易过冲发散）。消除「站着不动却有恒定小倾角」这种稳态误差，是**速度环**和**自校准中值**的活儿，见下文。

#### 环 2：速度环（VELOCITY loop）= PI —— 不让车「溜走」、能停住、能前进

**测量什么**：左右轮平均速度 `wheel_vel`（来自编码器）。
**输出什么**：`ax_velocity_out`，叠加到角度环输出上，**让车保持目标速度（默认 0 = 原地不动）**。
**为什么需要**：只有角度环时，车虽然不倒，但会**慢慢地朝某个方向滑行（drift）**永远停不下来 —— 因为「站直」和「停住」是两回事。地面有点斜、中值差一点点，车就会一直往前/往后溜。速度环的任务就是：**发现车在动，就轻轻改变车的目标倾角，让它自己刹住、回到原地。** 它也是「前进 / 后退」指令的执行者（把 `ax_robot_vx` 当目标速度）。

代码（`Balance_VelocityCtl()`，`ax_balance.c:129-157`）：

```c
if (t_vx > 700) t_vx = 700; else if (t_vx < -700) t_vx = -700;   // 目标速度限幅 ±700 mm/s

velocity_bias_update = 0 - r_vx;
velocity_bias = (velocity_bias * 0.6f) + (velocity_bias_update * 0.4f);   // 低通滤波 0.6/0.4

velocity_integral += velocity_bias * 0.01f;     // 对速度误差积分（10ms 一步）
velocity_target    = t_vx * 0.001f * 0.01f;
velocity_integral += velocity_target;           // 把目标速度也喂进积分（前进/后退）

if (velocity_integral >  1.5f) velocity_integral =  1.5f;        // 积分限幅 ±1.5（抗积分饱和）
else if (velocity_integral < -1.5f) velocity_integral = -1.5f;

p_out = -ax_velocity_kp * velocity_bias;        // 注意负号
i_out = -ax_velocity_ki * velocity_integral;    // 注意负号
out = p_out + i_out;
```

- **`ax_velocity_kp`（velocity Kp）**：比例增益，对「当前速度」起反应。车一动，P 项就给出一个反向纠正趋势，开始往回拉。**注意**：速度环并不直接「下达一个倾角」——它输出的是 PWM，被加到角度环输出上（见下文代码块后的段落），**间接地**改变小车的等效平衡点 / 平衡倾角，从而让车回来；它不会直接命令一个角度。Kp 太小刹不住、漂移明显；太大则车在原地**前后蹭来蹭去（低频来回）**。
- **`ax_velocity_ki`（velocity Ki）**：积分增益，对「速度误差的累积」起反应。它负责消掉**长期漂移**和**稳态位置误差** —— 哪怕只有一丁点持续的滑动，积分也会越攒越大，最终把车拽回来并**钉在原位**。Ki 太小则车会慢慢飘走、停不住；太大则**慢速来回振荡**、回位过冲。

> ### 重要：速度环是 **PI**，**没有** velocity Kd（不存在「vKd」）
> 你之前提到的 **「vKd」是一个误解** —— 固件里**根本没有速度微分增益**。看 `ax_robot.c:38-39` 只声明了 `ax_velocity_kp` 和 `ax_velocity_ki` 两个；`ax_control.c:28` 的可调增益数组也只有 `{ balance_kp, balance_kd, velocity_kp, velocity_ki }` 四个，蓝牙 `3` / `4` 分别选的是 **velocity Kp / velocity Ki**。速度环故意不用 D：轮速信号本身噪声大，对它做微分会把噪声放大成抖动；这里反而用了一个低通（0.6/0.4）去**平滑**速度。所以速度环只调两个数：**vKp、vKi**。

为什么 P 和 I 都带负号、为什么把目标速度喂进积分：因为速度环不是直接驱动电机，而是**通过改变车的等效平衡点**来影响速度。要让车前进，得让它**先往前倾一点**；负号 + 串级混合正好实现了「想往前走 → 制造一个往前的倾角 → 角度环去追这个倾角 → 车前倾 → 前进」的链条。

#### 环 3：转向环（TURN loop）= PD —— 控制左右转、保持直行

**测量什么**：偏航角速度 `turn_gyro`（陀螺仪 Z 轴）。
**输出什么**：`ax_turn_out`，一个**累加**的差速量，左右轮一加一减实现转向。
**为什么需要**：让车能听 `L`/`R` 指令转弯；不转弯时帮助车走直线、不自己跑偏。

代码（`Balance_TurnCtl()`，`ax_balance.c:160-175`）：

```c
bias  = t_vw - r_vw;                                  // 目标偏航速率 - 实际偏航速率
p_out = ax_turn_kp * bias * 0.001f;
d_out = ax_turn_kd * (bias - bias_last) * 0.001f;
out  += p_out + d_out;                                // 累加式输出
bias_last = bias;
```

- **`ax_turn_kp`（turn Kp）**：转向比例，决定转弯响应快慢。
- **`ax_turn_kd`（turn Kd）**：转向微分/阻尼，抑制转向时的过冲和摆头。

这是个 **PD** 环（无 I）。它的输出是 `out += ...` **累加**的：每个周期把偏航**速率误差**（`bias = t_vw - r_vw`）累加进 `out`。这相当于对偏航速率误差做了一个积分，效果是「**类航向（heading-like）**」控制 —— 但请注意它**不是**对一个真实绝对航向角的保持：累加的是**速率误差**而非积分得到的真实航向，而且 D 项是相邻两拍之差（会自相抵消/telescope），并且**一旦失能（`ax_robot_move_enable == 0`）`out` 立即清零**（`ax_balance.c:165-166`）。所以可以把它理解成「平时不下转向指令时把累计的偏航速率误差压住、帮助走直，下 `L`/`R` 时按速率转」，而不是一个会记住并回到某个绝对朝向的航向锁定。

### 1.4 三环如何混合 → 电机

三个环算完后混合（`ax_balance.c:80-86`）：

```c
wheel_pwm_l = ax_balance_out + ax_velocity_out - ax_turn_out;   // 左轮
wheel_pwm_r = ax_balance_out + ax_velocity_out + ax_turn_out;   // 右轮
AX_MOTOR_A_SetSpeed(wheel_pwm_l);
AX_MOTOR_B_SetSpeed(wheel_pwm_r);
```

直观理解：
- **角度环**对左右轮**同号**叠加 → 一起前后动来「接住」倾倒。
- **速度环**也对左右轮**同号**叠加 → 一起前后动来「停住 / 前进」。
- **转向环**对左右轮**异号**（`-` 和 `+`）→ 一边快一边慢，产生差速来转向。

**为什么三个都需要**：角度环让车「不倒」，速度环让车「不溜走、能定点、能前进」，转向环让车「能转向、走直线」。少了角度环 —— 立刻倒；少了速度环 —— 车不倒但满地乱溜停不住；少了转向环 —— 不能转、容易跑偏。

## 2. 机械中值（mechanical midpoint）

### 2.1 中值是什么、为什么重要

`ax_middle_angle`（`ax_robot.c:22`，单位 0.01°）是**机械平衡中点** —— 也就是车**真正能站稳、质心正好在轮轴正上方**时，DMP 读到的那个 pitch 值。

角度环的偏差是 `angle_bias = ax_middle_angle - angle`（`ax_balance.c:118`）。也就是说，**车不是去追「角度 = 0」，而是去追「角度 = 中值」**。

为什么重要：因为「DMP 读数为 0」**几乎肯定不等于**「车机械上站直」。由于电池、电路板、传感器安装并非完美对称，车真正的平衡姿态可能是 pitch = +1.3° 或 −0.8° 之类。如果你把中值设成 0，角度环就会拼命把车扶到「读数 0」那个**其实是歪的**姿态，结果车要么持续往一边使劲（然后靠速度环拉回来，来回较劲、抖），要么干脆站不住。**中值给错，再好的 Kp/Kd 也调不出干净的波形。**

### 2.2 DMP 的 pitch 是「重力绝对角」，不是「开机相对角」

一个关键点：**DMP 输出的 pitch 是相对重力的绝对倾角，不是相对开机姿态的角度。** DMP（Digital Motion Processor）在 MPU6050 内部把加速度计（感知重力方向）和陀螺仪融合，得到一个**长期不漂、以重力为基准**的姿态角。

实际意义：

- **不需要开机时把车摆直**。无论开机时车是斜的还是被你拿在手上，DMP 给的 pitch 都是「相对地面竖直方向」的真实倾角。
- 因此一个**固定的** `ax_middle_angle` 数值在每次开机后都代表**同一个真实姿态** —— 这正是「记录下来、下次硬编码直接用」能成立的根本原因（见第 7 节）。如果 pitch 是开机相对的，那记录的数字下次开机就没意义了。

### 2.3 自校准中值（self-calibrating midpoint）

固件有一个很贴心的功能：**在完整串级模式下、车正在平衡、且没有被指令移动时，它会让 `ax_middle_angle` 缓慢地朝车「真正的静止倾角」漂移**，自动收敛到正确的机械中值。

代码（`ax_balance.c:99-105`）：

```c
if (ax_velocity_enable && ax_robot_move_enable && ax_robot_vx == 0 && ax_robot_vw == 0)
{
    static int32_t mid_acc = 0;
    mid_acc += (ax_balance_angle - ax_middle_angle);   // 累积「当前实际角 - 当前中值」
    if (mid_acc > 4000)       { if (ax_middle_angle <  1500) ax_middle_angle++; mid_acc = 0; }
    else if (mid_acc < -4000) { if (ax_middle_angle > -1500) ax_middle_angle--; mid_acc = 0; }
}
```

原理：速度环会把车**定点钉住**，所以车最终会停在它**真正的平衡倾角**上。固件持续累积「实际角与当前中值之差」，一旦攒够阈值（4000），就把中值朝那个方向**挪一个最小步（1 个 0.01°）**，然后清零重来。结果是：

- 中值**自动收敛**到车真正的平衡姿态，「站着不动却有恒定小倾角」的稳态误差**自动消失**。
- 漂移**非常慢**（秒级），不会干扰平衡。
- 范围**限幅 ±15°**（`±1500`，即 `1500` / `-1500`），防止跑飞。
- **只在速度环开启 + 静止时**生效（纯角度环、或正在前进/转向时不动它）。

注意 `FallProtect`（倾角 > 45° 杀电机，`:217`）和 `PutDown`（在中值 ±5° 内被推一下就自动重启，`:190`）现在都是**相对 `ax_middle_angle`** 判定的，不是相对绝对 0，所以即便中值不为 0 也能正确工作。

**这一切的实用价值**：你不必手工把中值调到完美。开速度环（按 `V`），把车扶到大致竖直让它自己站一会儿，自校准会把中值收敛到位；然后你从 OLED 上读出这个收敛值，硬编码到固件里（第 7 节）。

## 3. 调参准备

### 3.1 烧录固件

把固件编译烧进 STM32F103C8T6。烧录后默认状态（来自源码）：

- `ax_velocity_enable = 0` 且 `ax_turn_enable = 0` → **只有角度环运行**，速度环和转向环输出被强制清零（`ax_balance.c`）。这是上电 / 角度环调试的安全状态。蓝牙 `V` 单独开速度环、`T` 单独开转向环 —— **一次只开一个**，方便隔离故障。
- 当前增益（`ax_robot.c`）：balance Kp=1200, Kd=80（你调好的值）；velocity Kp=8000, **Ki=0（上电安全值，验证速度环符号后再逐步加）**；turn Kp=1000, Kd=5000。厂商官方速度环是 8000/4000（配 Kp=2000），我们 Kp=1200，建议起调用更小的 **vKp≈4800 / vKi≈2400**。
- `ax_robot_move_enable = 1`（已使能）。

> 角度环已调到 **balance Kp=1200, Kd=80**（相对稳定），现在已是 `ax_robot.c` 的初值。调好的值可用蓝牙 **`W`** 存进片内 Flash（断电 / 重烧固件都保留），**`Z`** 擦除存档恢复代码初值。

### 3.2 连接 Android 蓝牙发实时指令（HC-05）

- 硬件：HC-05 蓝牙模块（接 USART2）。
- 手机装 **「Serial Bluetooth Terminal」**（Android），与 HC-05 配对、连接。
- 指令是**单个字符、大小写不敏感**，其它字节（CR/LF 等）被忽略（`ax_control.c`）。
- 这样你**不用反复烧录**就能在小车站着时实时改增益、切模式、捕获中值、驱动小车。

### 3.3 连接 CH340 + Arduino Serial Plotter 看四通道波形

- 硬件：USB-CH340 接 STM32 的 **USART1**，波特率 **230400**。
- PC 端用 **Arduino IDE 的 Serial Plotter**（或 SerialPlot），波特率设 230400。
- 固件在 `Control_Task` 里每 ~20 ms（约 50 Hz）打印一行**六个数**（`ax_task.c`）：

| 通道 | 含义 | 单位 | 看它干嘛 |
|------|------|------|---------|
| **ch1** | 角度误差（实际角 − 中值） | 0.01° | **0 = 平衡**。调 Kp/Kd 主要盯它 |
| **ch2** | 倾倒速率 tilt rate | deg/s | 看抖动/振荡的快慢 |
| **ch3** | 左轮速度 | mm/s | 编码器是否在计数（推轮子应有非 0 读数） |
| **ch4** | 右轮速度 | mm/s | 同上；恒 0 = 没接好 |
| **ch5** | 速度环输出 velocity_out | PWM | **判断速度环符号**：推轮子时它要**反向**（见下方） |
| **ch6** | 转向环输出 turn_out | PWM | 开转向环时看它是否积分爆走 |

> **速度环符号 = 看「刹车测试」，不要纠结 ch3/ch4 的正负（之前正是被 ch3/ch4 的正负带沟里反复翻车）。** 唯一判据：**速度环单独打开（`V` 开、`T` 关、vKi=0），用手拨轮子 → 电机必须「顶回来」(刹车)，ch5 与你拨的方向相反。** 如果轮子顺着你拨的方向**越跑越快**（ch5 飙到 ±1200 且和拨的方向一致），就是**正反馈** → 把 `ax_balance.c` 里两行 `ax_wheel_vel_l/ax_wheel_vel_r` 的符号**一起翻**（两行始终一正一负、左右镜像）。本车已按此测试确定为「刹车」方向。

> 注意：USART1 现在被波形流占用，固件**不再**用串口打印增益/状态文字；增益和模式都看 **OLED**（`ax_task.c:101-102`）。

### 3.4 蓝牙指令速查表（来自 `ax_control.c`）

| 键 | 作用 |
|----|------|
| `V` | 切换**速度环** on/off（单独验证速度环用） |
| `T` | 切换**转向环** on/off（**一次只开一个**；都关 = 纯角度环） |
| `1` | 选中 **balance Kp**（之后用 +/- 调） |
| `2` | 选中 **balance Kd** |
| `3` | 选中 **velocity Kp** |
| `4` | 选中 **velocity Ki** |
| `+`（或 `=`） | 把选中的增益**加**一步 |
| `-`（或 `_`） | 把选中的增益**减**一步 |
| `M` | 把**当前角度捕获为中值** `ax_middle_angle` |
| `F` / `B` | 前进 / 后退（目标 250 mm/s） |
| `L` / `R` | 左转 / 右转（目标 ~60 deg/s） |
| `S` | 停 |
| `O` | 切换超声波避障 |

**每按一次 +/- 的步长**（`ax_control.c:29`，对应 1/2/3/4 选中项）：

| 选中 | 步长 |
|------|------|
| `1` balance Kp | **50** |
| `2` balance Kd | **5** |
| `3` velocity Kp | **200** |
| `4` velocity Ki | **100** |

> 转向环增益 **turn Kp / turn Kd 不在蓝牙可调列表里**（数组只有 4 项），只能改源码 `ax_robot.c:43-44` 重烧。

### 3.5 OLED 布局（来自 `ax_task.c:65-97`）

```
L1:  V<0/1> T<0/1>  ARM/FALL       ← 速度环/转向环开关 + 使能(ARM)/跌倒(FALL)
L2:  A:<角度,deg>      M:<中值,0.01deg 原始值>
L3:  P:<balanceKp>     D:<balanceKd>
L4:  vP:<velocityKp>   vI:<velocityKi>
```

- **L1** 左：`V1`=速度环开 /`V0`=关，`T1`=转向环开 /`T0`=关（都为 0 即纯角度环）；右：`ARM` = 使能中，`FALL` = 已触发跌倒保护。
- **L2** `A:` 是当前倾角（deg）；`M:` 是中值**原始 0.01° 值**（这是你最终要抄下来硬编码的数）。
- **L3 / L4** 分别是角度环、速度环的增益，调参时实时变化、方便核对。

## 4. 一步一步调参

> 总思路（厂商 DB20 手册方法）：先 **I=D=0**，把 **Kp** 加到刚开始振荡，再加 **D** 压住振荡，最后加 **I** 消除稳态误差。判据看：**超调量 overshoot / 上升时间 rise time / 稳态误差 steady error**。下面按这个思路、结合本车三环顺序展开。

### Phase 0：安全准备

- **轮子先离地**（架空）做角度环上电检查。
- 确认 `ax_velocity_enable = 0` 且 `ax_turn_enable = 0`（OLED L1 显示 **`V0 T0`**）—— 此时只有角度环工作，速度/转向被清零，最安全。

### Phase 1：角度环 PD（ANGL 模式）

**目标**：车能稳稳立住，扶它一下能弹回竖直、不持续抖、不持续晃。

1. **先验证电机方向**（架空状态）。手把车往前倾，**两个轮子都应该往「前进 / 接住倾倒」方向转**；把车往后倾，两轮应反向。若某个轮转反了 —— 交换该电机的 AIN1/AIN2（或 BIN1/BIN2）接线，或翻转 `ax_balance.c:85-86` 里该 `SetSpeed` 的符号（`ax_balance.c:11-14, 21-23` 有说明）。**方向不对就调参，纯属白费。**

2. **按 vendor 方法调 Kp → Kd**：
   - 先把 **Kd 降到很小甚至 0**（蓝牙 `2` 选 Kd，按 `-`），只留 Kp。
   - 蓝牙 `1` 选 Kp，逐步 `+`（步长 50）**加大 Kp 直到车开始来回振荡 / 高频抖**。这说明 Kp 到了「刚够硬」附近，记下这个值。
   - 蓝牙 `2` 选 Kd，逐步 `+`（步长 5）**加 Kd 把振荡压下去**，直到扶一下能干脆地回正、过冲小。Kd 加过头会出现高频小抖、电机嗡嗒响 —— 退回去一点。
   - 反复微调 Kp/Kd，直到 **ch1（角度误差）扶动后能快速收敛回 0 附近、没有持续振荡**。

3. **你已有的起点**：balance **Kp=1200, Kd=80**（相对稳定）。可以从这里出发微调，不必从零开始。OLED L3 应显示 `P:1200 D: 80`。

4. **此阶段只看 ch1 / ch2**：ch1 是角度误差（要收敛到 0），ch2 是倾倒速率（抖动越小越好）。ch3/ch4 此时**先不管**（速度环还没开）。

> 提醒：角度环没有 I，所以**单独的角度环允许车整体缓慢漂移** —— 这正常，停住是速度环的活。本阶段只要求「不倒、扶动能回正、不持续抖」。

### Phase 2：确认编码器「在计数」（开速度环之前）

这一步只确认编码器**有没有在工作**；符号对不对放到 Phase 3 用「刹车测试」判（那才是唯一可靠的判据，别再盯 ch3/ch4 的正负——就是它把我们带沟里反复翻车的）。

- 车架空，看波形 **ch3（左轮速度）/ ch4（右轮速度）**。
- 用手转每个轮子：对应通道应有**明显非 0 读数**，正反转读数正负不同，松手回 0。
- 若某个轮**纹丝不动恒 0**：该编码器**没接好 / 没在计数**，这比接反更危险，先查接线 / 相序。本车编码器：enc A=PA15/PB3（TIM2），enc B=PB4/PB5（TIM3）（以编码器/TIM 初始化源码为准）。
- ch3/ch4 此刻是**正是负都不用管**——速度环的符号在 Phase 3 用「电机会不会刹车」来定。

### Phase 3：速度环 PI（按 `V` **单独**开速度环，转向环保持关）

**目标**：车不再漂移，能**定点钉住**，扶到任意起始位置都能站稳并停下；能听 `F`/`B` 前进后退。

> **第 0 步：用「刹车测试」定符号（仍架空！这是唯一可靠的判据）。** 蓝牙按 **`V`** 只开速度环（`T` 关；OLED `V1 T0`），`vKi=0`，只有 vKp。用手把轮子往任一方向拨 —— 电机必须**顶回来（刹车）**，ch5 与你拨的方向**相反**。✅ 顶回来 = 符号对，往下调增益。❌ 若轮子**顺着你拨的方向越跑越快**、ch5 飙到 ±1200 且与拨向一致 = **正反馈** → 把 `ax_balance.c` 里 `ax_wheel_vel_l` / `ax_wheel_vel_r` 两行符号**一起翻**（保持一正一负、左右镜像），重新编译再测。安全网：输出已被钳到 ±1200，所以这个测试即使符号反也不会像以前那样冲飞。

1. 符号确认无误后，**调 vKp**（蓝牙 `3` 选 velocity Kp，步长 200）：
   - 从较小值往上加。vKp 的作用是「一动就往回拉」。
   - 太小 → 车明显漂移、刹不住；太大 → 车在原地**前后蹭来蹭去（低频来回）**。加到「漂移基本被压住、但还没开始来回蹭」的程度。
2. **再调 vKi**（蓝牙 `4` 选 velocity Ki，步长 100；**从 0 开始逐步加**）：
   - vKi 消除**长期漂移和稳态位置误差**，把车**最终钉死在原位**。
   - 太小 → 车会缓慢飘走、停不干脆；太大 → **慢速来回振荡 / 回位过冲**。每加一档都看 ch5：静止时它不应慢慢爬向饱和。加到「松手后车会主动回到原点并稳稳停住」为止。
3. 反复在 vKp / vKi 之间微调，目标波形：**ch1 在 0 附近、ch3/ch4 静止时贴 0**，推一下能自己回来停住。

> 重申：速度环只有 **vKp、vKi 两个数**，**没有 vKd**。别去找速度微分 —— 它不存在（见第 1.3 节的红框）。

**速度环 + 自校准中值如何让小车「对起始位置鲁棒」**：
- 速度环把车**定点钉住**，于是车会停在它**真正的平衡倾角**上。
- 此时自校准中值（第 2.3 节，仅速度环开 + 静止时生效）持续把 `ax_middle_angle` 朝这个真实倾角缓慢收敛，**站立稳态误差自动归零**。
- 两者合力的结果：**你把车扶到大致竖直的任何起始姿态，它都能自己找到平衡点、停住、并把中值校准到位** —— 不需要你每次开机都精确摆放。这就是「鲁棒于起始位置」的来源。

### Phase 4：转向环 PD

**目标**：能干脆地左右转、转完不摆头；直行不跑偏。

- **先单独验证转向环**（架空）：蓝牙 `T` 单独开转向环（`V` 保持关，OLED `V0 T1`）。静止、不发转向指令时，ch6（turn_out）**不应自己越爬越大** —— 固件已加死区 + 累加器钳位 + 失能即清零来防爆走；用手轻扭车头，两轮应**差动反抗**这个旋转。若静止时 ch6 仍单调爬升，是 Z 轴陀螺零偏 / 符号反，先处理符号 / 零偏。
- turn Kp / turn Kd **不能用蓝牙调**（不在 4 项数组里），改源码 `ax_robot.c`（默认 turn Kp=1000, Kd=5000）重烧。
- 方法同 PD：先有 turn Kp 让它能响应 `L`/`R` 转向；不够灵就加 Kp，转弯摆头/过冲就加 turn Kd 阻尼。
- 验证 OK 后再同时开 `V`+`T`，用蓝牙 `L`/`R`/`S` 测试：转向是否干脆、停转后是否摆头、直行（不发转向指令时）是否走直。

## 5. 看波形 / 看 OLED 判读

### 5.1 好波形 vs 坏波形（Serial Plotter）

**ch1（角度误差，0 = 平衡）—— 调参主要看它：**

| 现象 | 解读 | 大致方向 |
|------|------|---------|
| 扰动后**快速、单调地回到 0**，几乎不过冲 | 理想 | 保持 |
| 回 0 但**明显过冲、来回几次才稳**（低频大摆） | 阻尼不足 | 角度环 Kd 偏小（或 Kp 偏大） |
| **持续高频小抖**、收不干净，电机嗡嗒响 | 噪声被放大 | 角度环 Kd 偏大（或 Kp 偏大） |
| **慢慢偏离 0 不回来**（纯角度环时） | 角度环无 I，正常漂移 | 开速度环 `V` 治 |
| 开速度环后 ch1 **慢速来回大摆** | 速度环回位过冲 | vKi 偏大 |

**ch2（倾倒速率）**：抖动越剧烈，ch2 振幅越大、越「毛」。干净的平衡 ch2 应在 0 附近小幅波动。

**ch3 / ch4（左右轮速）：**

| 现象 | 解读 |
|------|------|
| 转轮子有明显非 0 读数、松手回 0 | 编码器在计数（正常）；正负不用管，符号看 ch5 刹车测试 |
| 恒 **0**、转轮子不动 | 编码器没接 / 没在计数（Phase 2 查接线） |
| 静止时**持续同号偏置、越来越大** | 积分饱和（编码器问题或 vKi 过大） |
| 开速度环后静止 ch3/ch4 **来回小幅震荡** | 速度环来回蹭，vKp/vKi 偏大 |

### 5.2 OLED 判读

- **L1**：确认环开关对（Phase 1 应 `V0 T0`，Phase 3 验速度环时 `V1 T0`，Phase 4 验转向环时 `V0 T1`）；若右侧变 **`FALL`** 说明触发了 45° 跌倒保护（已杀电机），把车扶到中值 ±5° 内并轻推一下会自动重启（`PutDown`）。
- **L2 `A:`**：当前倾角，平衡时应贴近中值对应角度；`M:` 是中值原始值，速度环开 + 静止时会看到它**缓慢自校准移动**直到稳定。
- **L3 / L4**：实时核对你正在调的增益值是否如预期变化（每按一次 +/- 跳一个步长）。

## 6. 常见问题对症表（症状 → 调哪个增益）

| 症状 | 最可能原因 | 怎么改 |
|------|-----------|--------|
| 车直接倒、扶不住 | 角度环 Kp 太小 / 电机方向反 / 中值离谱 | 先验电机方向(Phase1.1)，加 balance Kp，检查中值 |
| 站着**高频抖**、电机嗡嗒响 | 角度环 Kd 太大（或 Kp 太大） | 减 balance Kd（按 `2` 再 `-`）；必要时减 Kp |
| 扶动后**过冲、低频来回摆**几下才稳 | 角度环 Kd 太小 | 加 balance Kd |
| 整体太「软」、外力一推就倒 | 角度环 Kp 太小 | 加 balance Kp |
| 纯角度环慢慢漂走停不下 | 正常（角度环无 I） | 开速度环 `V`（Phase 3） |
| 开速度环后仍**朝一个方向漂移**停不住 | 速度环 vKp/vKi 太小 | 先加 vKp，再加 vKi |
| 开速度环后原地**前后蹭来蹭去（快）** | 速度环 vKp 太大 | 减 velocity Kp |
| 开速度环后**慢速来回大摆 / 回位过冲** | 速度环 vKi 太大 | 减 velocity Ki |
| **一开速度环就冲飞 / 顺手一拨就越跑越快、ch5 飙 ±1200** | **速度环符号反**（正反馈，电机助力而非刹车） | Phase 3「刹车测试」：把 `ax_balance.c` 两行编码器符号**一起翻** |
| 两轮无视倾角**死顶一个方向**、冲飞 | 编码器没接 / 恒 0，积分饱和 | 回 Phase 2 验编码器；验证前别开速度环 |
| **一开转向环就原地打转 / 越转越快** | 转向环累加器爆走 / Z 陀螺零偏 | 单独 `T` 验证；已加死区+钳位，仍爬则查陀螺符号/零偏 |
| 转弯**摆头 / 过冲** | turn Kd 太小 | 改源码加 turn Kd 重烧 |
| 转弯**迟钝 / 转不动** | turn Kp 太小 | 改源码加 turn Kp 重烧 |
| 不转弯也**自己跑偏** | turn 环偏弱 | 加 turn Kp/Kd（源码） |
| 站着不动却有**恒定小倾角** | 中值不准 | 开速度环(`V`)让自校准收敛，或按 `M` 手动捕获 |

> **总口诀**：抖 = 看 Kd（太大高频抖、太小过冲摆）；硬不硬 = 看 Kp；漂不漂、停不停 = 看 vKp/vKi；转向 = 看 turn Kp/Kd（源码）。

## 7. 记录与下次直接用

调好之后，把收敛的数值**从 OLED 读出来、硬编码进固件**，下次烧录即用、不必每次重调。

### 7.1 从 OLED 读出最终值

- **L2 `M:`** → 收敛后的中值原始值（0.01°）。
- **L3 `P:` / `D:`** → 最终 balance Kp / Kd。
- **L4 `vP:` / `vI:`** → 最终 velocity Kp / Ki。
- turn Kp/Kd 不在 OLED 上 —— 用你源码里的当前值即可。

### 7.2 硬编码到 `ax_robot.c`

打开 `self-balancing-cart/STM32/Robot/ax_robot.c`，把下面这些变量的初值改成你读到的收敛值：

| 变量 | 位置 | 说明 | 例（你的当前/起点值） |
|------|------|------|---------|
| `ax_middle_angle` | `ax_robot.c:22` | 机械中值（0.01° 原始值，抄 OLED 的 `M:`） | 抄 OLED |
| `ax_balance_kp` | `ax_robot.c:33` | 角度环 Kp（抄 OLED `P:`） | `1200` |
| `ax_balance_kd` | `ax_robot.c:34` | 角度环 Kd（抄 OLED `D:`） | `80` |
| `ax_velocity_kp` | `ax_robot.c:38` | 速度环 Kp（抄 OLED `vP:`） | 收敛值 |
| `ax_velocity_ki` | `ax_robot.c:39` | 速度环 Ki（抄 OLED `vI:`） | 收敛值 |
| `ax_turn_kp` | `ax_robot.c:43` | 转向环 Kp | `1000` |
| `ax_turn_kd` | `ax_robot.c:44` | 转向环 Kd | `5000` |

例如把 `ax_robot.c:33-34` 改成：

```c
int16_t  ax_balance_kp = 1200;   /* tuned */
int16_t  ax_balance_kd = 80;     /* tuned */
```

中值改 `ax_robot.c:22`：

```c
int16_t  ax_middle_angle = 130;  /* 例：OLED 上 M: 读到的收敛值（0.01deg）*/
```

> **小贴士**：硬编码中值后，下次开机就直接站在正确平衡点上（因为 DMP pitch 是重力绝对角，第 2.2 节），自校准只需做很小的微调甚至几乎不动。

### 7.3 未来可加：内部 Flash 自动保存

目前增益和中值**断电即丢**，每次开机回到 `ax_robot.c` 里的初值。一个**未来可加的功能**是：把收敛后的增益/中值写进 STM32 的**内部 Flash**（或加片外 EEPROM），开机时读回，省去手工抄写硬编码。这是一个明确的、可选的后续改进项 —— 目前的做法仍是手工硬编码。

## 8. 卡尔曼滤波说明

经常有人问「是不是加个卡尔曼滤波（Kalman filter）就不抖了」。针对本车，结论很明确：

### 8.1 DMP 已经做了传感器融合，在它输出上再套卡尔曼是**多余的**

本固件的角度来自 **MPU6050 的 DMP**（`AX_MPU6050_DMP_GetData`，`ax_balance.c:52`）。DMP 在芯片内部**已经把加速度计和陀螺仪融合**成一个稳定、低漂移、以重力为基准的姿态角（pitch `angle[1]`）。卡尔曼滤波 / 互补滤波的**作用本来就是做这件融合** —— 既然 DMP 已经做完了，**在 DMP 的输出角上再叠一个卡尔曼，是重复劳动**，只会增加延迟、几乎不会改善什么。

### 8.2 卡尔曼/互补滤波是 DMP 的**替代方案**，不是叠加方案

如果你**不用 DMP**，而是直接读**原始**加速度 + 原始陀螺，然后**自己用卡尔曼或互补滤波融合**得到角度 —— 这是一条**可选的替代路线**。它的优点是你对融合过程有**完全掌控、延迟更低**（DMP 内部有固定流水线延迟）；缺点是这是一次**大改动**：要重写姿态解算、重新标定、重新整定整个角度环，风险和工作量都不小。它和 DMP 是**二选一**，不是「在 DMP 之上再加」。

### 8.3 当前的抖动是 **PID / 中值问题，不是传感器噪声问题**

最关键的一点：**你现在看到的抖动，根因是 PID 整定或中值，不是传感器噪声。** DMP 给的角度已经相当干净。本指南第 5、6 节里描述的抖/摆，几乎都能对应到某个增益（balance Kd 过大 → 高频抖、过小 → 过冲摆；中值不准 → 持续偏置较劲）或编码器/速度环问题。**所以加卡尔曼治不好当前的抖** —— 正确做法是按第 4、6 节调 Kp/Kd/vKp/vKi 和校准中值。先把 PID 和中值调对，再谈要不要换姿态解算方案。

## 9. 关于自动调参

诚实地讲清楚哪些能自动、哪些不建议自动：

### 9.1 自动找**中值** —— 已实现 ✅

自动寻找机械中值这件事，固件**已经做了**：就是第 2.3 节的**自校准中值**。在速度环开 + 静止时，它会让 `ax_middle_angle` 缓慢收敛到车真正的平衡倾角。这部分**自动化是成熟、可靠、已经在用**的。

### 9.2 自动整定 **PID 增益** —— 不建议 ⚠️

自动整定 **PID 增益（Kp/Kd/vKp/vKi）** 则完全是另一回事。在一台**会摔倒**的真实硬件上自动调 PID 是**有风险、属于研究级**的难题：

- 自动整定算法（如继电反馈 relay feedback、各种自适应/极限搜索）通常需要**故意把系统推到振荡边缘**去探边界 —— 在倒立摆上，这等于**故意让车反复接近摔倒**，很容易真摔、撞坏硬件。
- 它需要可靠的实验复位、安全限位、失败检测，工程量大且仍不保证稳定收敛。
- 对一台课程/原型小车来说，性价比很低、风险偏高。

**因此不推荐**在本车上做硬件 PID 自动整定。**实用、可靠的路径就是本指南第 4 节的手动整定流程**：vendor 的「Kp→Kd→Ki」方法 + 蓝牙实时调 + 串口波形判读 + OLED 读值硬编码。中值交给已实现的自校准，增益靠手调 —— 这是当前最稳妥的组合。
